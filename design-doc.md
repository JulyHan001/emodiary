# 情感日记智能体助手 — 设计文档

## 1. 项目概述

**项目名称**: EmoDiary — 情感日记 + 成长追踪 AI 智能体

**一句话描述**: 用户记录每日情绪和想法，AI 智能体识别情绪、存储日记、检索历史，生成个性化成长洞察报告。

**项目目标**:
- 作为 Agent 开发的学习项目，覆盖 Prompt Engineering、Agent 架构、RAG 三大核心技术
- 2 周内完成可部署版本，上传 GitHub 作为求职作品集
- 面试时能清晰讲解每一层架构决策的"为什么"

---

## 2. 核心功能

| 功能 | 描述 | 涉及技术 |
|------|------|----------|
| 情绪对话 | 用户用自然语言记录心情，AI 识别情绪类别和强度 | Prompt Engineering |
| 日记存储 | 结构化提取日记内容（情绪标签、关键事件、日期） | Structured Output |
| 历史检索 | "我上周心情怎么样？" → 检索相关日记条目 | RAG (向量检索) |
| 成长报告 | 基于历史数据生成周/月情绪趋势分析 | Agent 编排 + RAG |
| 建议生成 | 根据情绪模式，给出个性化的心理调节建议 | RAG (心理学知识库) |

## 4. 技术栈详解

| 层 | 技术选型 | 选型理由 |
|----|---------|---------|
| LLM | OpenAI GPT-4o-mini / Claude Haiku | 成本低、速度快，适合日记场景 |
| Agent 框架 | LangGraph | 状态机模式，可视化工作流，面试热门话题 |
| 向量数据库 | ChromaDB | 纯 Python，零配置，本地运行，新手友好 |
| 结构化存储 | SQLite | 无需安装，Python 内置，存储日记元数据 |
| API | FastAPI | 自动生成 API 文档，async 支持，行业标准 |
| 前端 | Streamlit | Python 原生 UI，几行代码出页面，适合原型 |
| 部署 | Docker + Railway/Render | 免费额度，一键部署，适合作品展示 |

## 5. 数据模型

```python
class DiaryEntry:
    id: str              # UUID
    user_id: str         # 用户标识
    content: str         # 原始用户输入
    emotion: str         # 主要情绪: happy/sad/anxious/angry/calm/excited

```

## 5.3 RAG 技术选型决策

### Embedding 模型选择

| 考量 | 决策 | 理由 |
|------|------|------|
| 开发阶段 | OpenAI text-embedding-3-small | API 调用简单，一行代码，专注学 Agent 架构 |
| 进阶/优化 | BGE-M3 (BAAI 智源) | 中文最优、免费本地推理、面试加分项 |

**为什么不一开始就用 BGE-M3？**
BGE-M3 需要本地加载模型（~2GB），涉及 PyTorch/sentence-transformers 环境配置。新手阶段先用 OpenAI API 跑通 RAG 全链路，理解原理后再切换本地模型——这本身就是一个很好的"优化迭代"故事。

**面试话术**: "第一版用 text-embedding-3-small 快速验证 RAG pipeline，第二版切 BGE-M3 因为：(1) 原生中文表现更好 (2) 零 API 成本 (3) C-MTEB 排名靠前"

### 相似度计算选择

| 方法 | 公式 | 本项目选择 |
|------|------|-----------|
| **余弦相似度** ✅ | cos(θ) = A·B / (\|A\| × \|B\|) | **选这个** |
| 欧氏距离 | d = √(Σ(ai - bi)²) | 不选：受文本长度影响 |
| 点积 | A·B = Σ(ai × bi) | 不选：需要归一化前提 |

**选择理由**: 日记长短不一（"今天好累" vs 一大段），余弦只比较语义方向不比长度，最适合文本语义检索。ChromaDB 默认支持。

### 向量索引结构选择

| 索引 | 原理 | 精度 | 速度 | 适用规模 |
|------|------|------|------|---------|
| **Flat (暴力)** ✅ | 逐个比较 | 100% | O(n) | < 10K |
| HNSW | 多层近邻图 | 95-99% | O(log n) | 10K-10M |
| IVF | 聚类分区 | 90-95% | 中等 | > 100K |

**选择理由**: 个人日记数据量小（几百到几千条），暴力搜索精确且延迟 < 1ms。ChromaDB 底层用 HNSW，在小数据量下行为等同于 Flat，无需调优。

**扩展方案**: 若扩展到多用户百万条日记 → 切换 Qdrant + HNSW + 分区索引。

### 向量数据库选择

| 数据库 | 部署 | 学习曲线 | 本项目适配度 |
|--------|------|---------|------------|
| **ChromaDB** ✅ | pip install | ⭐ 最低 | 最佳——零配置、Python 原生 |
| FAISS | 本地库 | ⭐⭐ | 纯索引库，无持久化管理 |
| Qdrant | 本地/云 | ⭐⭐ | 生产级备选方案 |
| Milvus | 分布式 | ⭐⭐⭐ | 过重，不适合学习项目 |
| Pinecone | 纯云 | ⭐⭐ | 收费，且失去本地调试能力 |

**选择理由**:
1. `pip install chromadb` 零配置启动
2. 默认余弦相似度 + HNSW，无需手动配索引
3. 数据持久化到本地目录，重启不丢失
4. 面试时说："选 ChromaDB 做原型验证，生产环境可切 Qdrant（Rust 性能）或 Milvus（分布式）"

## 6. Agent 工作流设计 (LangGraph 状态机)

### 6.1 日记记录流程

```
用户输入
    │
    ▼
[emotion_detect] ──── 识别情绪类别和强度
    │
    ▼
[structured_extract] ── 提取关键词、生成摘要
    │
    ▼
[store] ──────────── 写入 SQLite + ChromaDB
    │
    ▼
[respond] ─────────── 生成共情回复 + 可选建议
```

### 6.2 历史查询流程

```
用户提问 ("我这周心情怎么样？")
    │
    ▼
[intent_classify] ──── 判断: 查询历史 / 生成报告 / 闲聊
    │
    ├──▶ [rag_search] ──── 向量检索相关日记
    │        │
    │        ▼
    │    [synthesize] ──── 基于检索结果生成回答
    │
    └──▶ [report_gen] ──── 聚合数据生成趋势报告
```

## 7. Prompt 设计规范

### 7.1 情绪识别 Prompt

```
你是一个情绪分析专家。分析用户的文字，识别其主要情绪。

规则:
1. 从以下类别中选择最匹配的情绪: happy, sad, anxious, angry, calm, excited, confused
2. 给出情绪强度 (0.0-1.0)
3. 提取 2-5 个关键词

输出格式 (JSON):
{
  "emotion": "anxious",
  "score": 0.7,
  "keywords": ["工作压力", "deadline", "睡不好"],
  "summary": "因为工作deadline临近感到焦虑，影响了睡眠"
}

用户输入: {user_input}
```

### 7.2 共情回复 Prompt

```
你是一个温暖、有同理心的情感助手。根据用户的情绪分析结果，生成一段回复。

规则:
1. 先共情（"我理解你的感受..."）
2. 再提供一个具体、可执行的小建议
3. 语气温和、不说教
4. 控制在 100 字以内

情绪分析: {emotion_analysis}
用户原文: {user_input}
```

### 7.3 RAG 检索 + 报告生成 Prompt

```
基于用户的历史情绪日记数据，生成一份成长洞察报告。

检索到的相关日记:
{retrieved_entries}

分析要求:
1. 情绪趋势: 这段时间整体情绪走向
2. 关键事件: 哪些事件明显影响了情绪
3. 模式识别: 是否有重复出现的情绪触发因素
4. 成长建议: 基于模式给出 1-2 条可执行建议

输出格式: 使用 markdown，包含标题和分点
```

## 10. 面试展示要点

当面试官问到这个项目时，你可以按这个结构讲：

1. **为什么做这个？** → 选择"情感日记"场景是因为它天然需要多轮对话、结构化输出、历史检索三种能力，能完整覆盖 Agent 开发核心技术栈

2. **架构怎么设计的？** → LangGraph 状态机编排多个专用 Agent 节点，每个节点职责单一；双存储架构（SQLite 存结构化数据 + ChromaDB 存向量）解决"精确查询"和"语义检索"两种需求

3. **Prompt 怎么设计的？** → 情绪识别用结构化输出保证格式一致；共情回复用角色设定 + 规则约束保证语气；报告生成用 few-shot 示例保证质量

4. **RAG 怎么做的？** → 用户日记 embedding 后存入 ChromaDB，查询时先向量检索相关条目，再传给 LLM 生成基于真实数据的回答，避免幻觉

5. **遇到什么挑战？** → (在开发过程中记录，这是面试加分项)
